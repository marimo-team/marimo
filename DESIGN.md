## Intent

marimo UI should feel compact, calm, readable, and native to a reactive notebook. Prefer clear controls, quiet surfaces, precise spacing, and useful output over decorative gradients or marketing layouts.

## Token Use

The YAML frontmatter is generated from marimo frontend tokens and layout defaults. Component entries are intentionally small: they are canonical examples, not a reverse-engineered inventory of every UI primitive.

Use related token families for variants: hover, active, dark, destructive, error, success, stale, and visited states should derive from the same color families as their base component.

## Brand Assets

- Logo SVG: https://raw.githubusercontent.com/marimo-team/marimo/main/docs/_static/marimo-logotype-thick.svg
- Preserve the original aspect ratio.
- Do not recolor unless explicitly requested.

## Components

Buttons should use compact heights, label typography, md radius, and the primary, secondary, or action color families. Inputs should be compact, bordered, readable, and use code typography for code-like values.

Notebook cells, outputs, code editors, and data views should be full-width, overflow-safe, and visually quieter than the controls around them.

## UI Guidance

- Use PT Sans for UI and prose, Lora for authored markdown headings, and Fira Mono for code-like values.
- Keep controls compact and use generated tokens for sizing, radius, and semantic colors.
- Use borders first and subtle shadows second. Avoid floating section cards and nested cards.
- Reserve wider layouts for grids, dashboards, slides, and apps that need comparison or scanning.

## Motion

Use short, functional motion for state changes, loading, focus, and drag interactions. Avoid decorative animation.
